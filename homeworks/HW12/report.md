# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Для решения использовался датасет `S12-hw-dataset.csv` с колонками `date` и `target`; прогнозировалось следующее значение временного ряда.
- Данные были отсортированы по времени и разбиты на `train / validation / test` только хронологически, без `random split`.
- Были сравнены четыре обязательных подхода: `B1` (`naive-last`), `B2` (`moving-average`), `B3` (`ridge-lag-features`), `R1` (`gru-forecast`).
- Лучшим по `validation MAE` оказался `B3` (`Ridge` на лаговых, rolling- и календарных признаках), поэтому именно он был один раз оценен на `test`.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: `1 час`
- Горизонт прогноза: `1 шаг вперед`
- Размер датасета: `4320` наблюдений
- Диапазон дат: `2025-01-01 00:00:00` - `2025-06-29 23:00:00`
- Есть ли дополнительные внешние признаки: нет, использовался только сам ряд и признаки, построенные из прошлого
- Комментарий (2-4 предложения): В данных нет пропусков, поэтому основной акцент был на корректной работе со временем и признаками без утечки. Ряд выглядит нестационарным: заметны колебания уровня и локальные всплески. Так как ряд почасовой, полезными оказываются как короткие лаги, так и признаки суточной и недельной структуры. Это делает задачу удобной для сравнения простых baseline-моделей и рекуррентной модели.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: по хронологическому порядку строк после сортировки по `date`
- Какие доли или интервалы использовались: `70% / 15% / 15%`, то есть `3024 / 648 / 648` строк
- Почему `random split` здесь некорректен: при случайном перемешивании будущие наблюдения попадают в обучение, что создает утечку информации и завышает оценку качества

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: `lag_1`, `lag_7`, `lag_14`, а также дополнительные `lag_24`, `lag_168`
- Какие rolling-признаки использовались: `rolling_mean_7`, `rolling_std_7`, `rolling_mean_24`, `rolling_std_24`
- Какие календарные признаки использовались: `hour`, `dayofweek`, `dayofmonth`, `month`, `is_weekend`
- Как обрабатывались пропуски после построения лагов/rolling: строки с `NaN`, появившиеся только из-за сдвигов и окон, были отброшены
- Использовалось ли масштабирование и где именно оно обучалось: `StandardScaler` обучался только на `train` для `B3`; для `GRU` scaler обучался только на `train`-части целевого ряда

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): прогноз равен последнему известному значению (`lag_1`)
- B2 (`moving-average`): прогноз равен среднему по предыдущим `24` наблюдениям
- B3 (`ridge-lag-features`): `Ridge(alpha=1.0)` на лаговых, rolling- и календарных признаках
- R1 (`gru-forecast`): `GRU(hidden_size=64, num_layers=1)` на окнах длины `48` по самому ряду `target`

Дополнительно:

- Основная метрика выбора лучшей модели: `validation MAE`
- Дополнительные метрики: `RMSE`, `MAPE`
- `window_size` для `GRU`: `48`
- Batch size: `64`
- Максимальное число эпох: `30`
- Optimizer: `Adam`
- Learning rate: `0.001`
- Как сохранялась лучшая модель: по лучшему `validation MAE`; сохранялись `state_dict()` в `./artifacts/best_gru.pt` и конфиг в `./artifacts/best_gru_config.json`

## 7. Анализ

Наивный baseline `B1` оказался не таким слабым, потому что в почасовых рядах соседние значения часто сильно коррелируют между собой. Однако одного последнего значения недостаточно, чтобы уловить более длинные повторяющиеся шаблоны и изменения уровня ряда. `Moving average` с окном 24 сглаживает ряд слишком сильно, из-за чего хуже реагирует на резкие локальные изменения и дает самую высокую ошибку среди всех подходов. Лаговые и rolling-признаки позволили явно передать модели информацию о краткосрочной динамике, локальной дисперсии и календарной структуре, поэтому `Ridge` оказался лучшим. Это хороший пример того, что для умеренно сложного ряда сильный табличный baseline может быть очень конкурентоспособным.  

`GRU` помогла автоматически извлекать зависимость из окна прошлых значений и показала качество, близкое к лучшему baseline. При этом в данном датасете табличные признаки уже хорошо описывают структуру ряда, поэтому рекуррентная модель не дала явного выигрыша. Потенциальные утечки в такой задаче возникают при использовании `random split`, при rolling без `shift(1)` и при обучении scaler на всех данных сразу. Выбор лучшей модели по `validation`, а не по `test`, принципиален: иначе test перестает быть честной финальной оценкой. Наиболее показательные ошибки модели видны на локальных скачках и участках, где текущий уровень ряда быстро меняется.

## 8. Итоговый вывод

В качестве базового решения для такого ряда разумно брать табличную модель на лаговых и rolling-признаках, потому что она дает высокое качество, проста в интерпретации и быстро обучается. Простого baseline бывает достаточно, если структура ряда хорошо объясняется ближайшей историей и календарными признаками. Переход к `GRU` и другим нейросетям имеет смысл, когда ряд длиннее, сложнее и в нем важны нелинейные зависимости, которые трудно задать вручную. Главный вывод из работы состоит в том, что для временных рядов корректная валидация по времени важнее красивой модели: без нее даже сильный результат нельзя считать надежным.
